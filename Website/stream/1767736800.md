# Aftermath

There are a decent amount of issues when encoding annoying to use hypotheses around the instructions that need it. For example a `(h : (b : iN n) < n)` is quite unwieldy for the user and proving side.

- [ ] Encode the each hypothesis in a better way. For example you could expose a standard set of propositions (like `Pow2` for `iN`, `LtWidth b` for the hypothesis above, etc.)

- For example `Pow2` over `iN` could be proven if you're plugging in the value of a shift into `Pow2`.
- Focus on not exposing a Lean specific encoding of the hypotheses, for example, instead of `n ≥ 2` write `WGt 2` or `WidthGt 2`.

- Each hypothesis can become a decidable instance as well.
